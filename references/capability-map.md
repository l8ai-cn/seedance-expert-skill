# Capability Map — plan from scoped evidence and observed limits

Use this map before prompt planning. Labels: [official] means retained provider documentation within its stated model, surface, operation, region, and date; [field] means a scoped practitioner report; [heuristic] means a production starting point to test. Model capability, product access, request syntax, and returned adherence are separate layers.

## Audio and multi-shot boundary map

Resolve all eight rows for the selected request. Unknown surface or operation support fails closed.

| Boundary | Retained status | Safe planning consequence |
|---|---|---|
| Model audiovisual capability | a retained model-level claim may describe broad modalities without defining product access | cite only its exact scope; select a surface before compiling |
| Surface audio input | operation-specific; an upload control or accepted media type is required | bind one reference to one explicit target role using the selected profile |
| Surface audio output | operation-specific and independent from audio input | verify whether the result contains audio before promising a sound-on deliverable |
| Dialogue | no retained universal exact-dialogue guarantee | require resolved speaker, exact utterance, spoken language, ordered turn, and subtitle policy |
| Lip-sync | no retained universal control or adherence guarantee | treat mouth movement, audio, words, speaker, and timing as separate checks |
| Voice reference | authorization and accepted input role are necessary but not sufficient | do not promise cloning, exact replay, direct lip-sync, or speaker fidelity |
| Multi-shot grammar | provider examples can conflict on labels and timing, and an example is not an adherence guarantee | choose the selected operation's evidenced renderer; never assume universal labels or ranges |
| Adherence | every returned take is an observation, not proof of internal mechanics | define and score cut, event, timing, picture, dialogue, sync, and mix checks separately |

The next versioned audio/multi-shot contracts should carry these scope fields and semantic requirements into deterministic validation. Until an active checker accepts them, do not flatten dialogue or multi-shot state into V7-07 or claim compiler provenance for a hand-edited prompt.

## Candidate tests, not capabilities

- Multi-shot: author ordered semantic shots and typed transitions, then render only through an evidenced surface/operation grammar. Provider examples do not guarantee cut or range adherence.
- Audio-bearing output: request named observable sound and review the result. Do not rename this `native synced audio` without a scoped output claim and measured synchronization.
- Beat alignment: where an operation accepts the required audio binding, test one named cue against one visible event. This does not prove an audio clock or internal audiovisual mechanism.
- Voice input: where supported and authorized, test one narrow role. Accepted input does not prove exact playback, cloning, speaker identity, or lip-sync.
- Motion/reference transfer: specify the donor's exact target dimension and exclusions. Do not assume media type or upload order chooses authority.

## Design around observed limits

For connected generations, keep each clip small, record accepted observed state, preserve exact target/dimension authority, and re-anchor when a named continuity check fails. A project may declare its own review or re-anchor cap, but the skill does not impose a universal chain-depth schedule.

Surface duration caps are active-operation facts. Audio continuity across separate calls needs verification, so plan a unifying score in post when needed. Put fragile on-screen text in post when the selected workflow needs a clean plate. Prefer positive observable states to long defect lists [heuristic]. Review distant faces, hands, logos, and text closely [field]. Stage facial performance with readable blocking [heuristic]. Re-anchor from authorized canonical references when drift is observed [field]. Simplify crowded character-to-prop contact [heuristic]. Tier and seed behavior are surface-specific; neither is a fidelity guarantee.

## Competitive context

Do not market a capability as unique from an undated comparison table. Describe only what current retained evidence supports for the selected model, surface, operation, region, and date. Resolution, duration, accepted references, audio input, audio output, dialogue, lip-sync, voice behavior, multi-shot grammar, and adherence remain distinct claims; check `api-status.md` and the active profile before making a delivery promise.
